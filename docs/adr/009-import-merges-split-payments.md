# ADR-009: Import Excel ต้องประกอบรายจ่ายที่ตัดหลายใบเสร็จกลับเป็นรายการเดียว

สถานะ: ยอมรับแล้ว (21 ก.ค. 2569)

## บริบท

Session ก่อนหน้าต้องไล่แก้ข้อมูลจริงใน production DB ด้วย SQL มือ **7 ชุด** เพราะพบว่า
"รายจ่ายรายการเดียวกัน" ถูกบันทึกกระจายเป็นหลายแถวใน `expenses` โดยใช้เลขที่ส่งออก (`doc_no`) ซ้ำกัน
ตอนนั้นสรุปสาเหตุไว้ว่าผู้ใช้กด "+เพิ่มรายจ่าย" แยกทีละใบเสร็จแทนที่จะเพิ่มครั้งเดียวแบบตัดหลายใบ
และยังเหลืออีก 14 ชุดที่ยังไม่ตัดสินใจ (11 ชุดยืนยันแล้วว่าเป็นเอกสารรวมหลายรายการจริง ไม่ใช่บั๊ก)

Tech-debt audit เชิงรุกรอบถัดมาพบว่านั่นไม่ใช่สาเหตุเดียว — **Excel importer เองสร้างข้อมูลรูปนี้
โดยอัตโนมัติทุกครั้งที่ import** ไฟล์รายจ่าย Excel มีโครงสร้าง 1 sheet ต่อ 1 ใบเสร็จ
(ดู [ADR-002](002-expense-allocation-model.md)) ดังนั้นรายจ่ายที่จ่ายครั้งเดียวแต่ตัดเงินจาก
2 ใบเสร็จจะปรากฏเป็น "แถวใน sheet A" + "แถวใน sheet B" คนละแถว — `parseWorkbook()` (เดิม) แตกทุก
แถวเป็น `ParsedExpense` แยกกันหมด แล้ว import ทีละแถวผ่าน `save_expense` เป็นคนละ expense row
ได้ N รายจ่าย × 1 allocation แทนที่จะเป็น 1 รายจ่าย × N allocations ตามโมเดลที่ ADR-002 ตั้งใจไว้

ADR-002 เคยบันทึกพฤติกรรมนี้ไว้ในหัวข้อ "ผลที่ตามมา" ว่า *"ข้อมูลเก่าที่ import จากไฟล์รายจ่าย
(1 sheet ต่อใบ) จะกลายเป็นรายจ่าย 1 รายการต่อ 1 ใบเสมอ (allocation เดียว)"* — ยอมรับไว้เป็นข้อจำกัด
ของข้อมูลเก่า แต่ที่จริงคือข้อมูลผิดรูปตามโมเดลของ ADR-002 เอง และเกิดซ้ำได้กับการ import ในอนาคตด้วย
ไม่ใช่ปัญหาเชิงประวัติศาสตร์ — **ADR นี้ supersede เฉพาะบรรทัดนั้นของ ADR-002**

## การตัดสินใจ

เพิ่มขั้นตอน `groupExpenseSplits()` ใน [`import-excel.ts`](../../src/lib/import-excel.ts)
รันหลัง parse แถวดิบ ก่อนคืนผลเป็น `ParseResult.expenseGroups` — รวมแถวเป็นรายจ่ายเดียวเมื่อ
`doc_no` (ไม่ว่าง) + `description` + `paid_date` ตรงกันทุกตัวอักษรข้าม sheet เท่านั้น:

| กรณี | การกระทำ |
|---|---|
| `doc_no`+`description`+`paid_date` ตรงกัน คนละ `receipt_no` | รวมเป็น 1 รายจ่าย หลาย allocation |
| key เดียวกัน `receipt_no` เดียวกัน (แถวซ้ำใน sheet เดิม) | รวม amount เป็น allocation เดียว |
| `doc_no` ตรงกันแต่ `description` ต่างกัน | **ไม่รวม** — แจ้งเตือนให้ผู้ใช้ตรวจสอบเอง |
| `doc_no` ว่าง | **ไม่รวมเด็ดขาด** (ไม่มีหลักฐานพอ) — เตือนถ้า description+paid_date ซ้ำข้าม sheet |

หน้า import ([settings/import/page.tsx](../../src/app/(app)/settings/import/page.tsx)) วนตาม
`expenseGroups` แทน `expenses` ดิบ, resolve ใบเสร็จทุกใบของกลุ่มก่อน insert (ขาดใบไหนล้มทั้งกลุ่ม
กันได้ข้อมูลตัดไม่ครบใบกลับมาอีก), และแสดง preview "พบ N แถว → รวมเป็น M รายจ่าย" ก่อนกดนำเข้าจริง

ฝั่งกรอกมือ ([expense-form.tsx](../../src/components/expense-form.tsx)) เพิ่มคำเตือนไม่บล็อก
เมื่อ `doc_no` ที่กรอกซ้ำกับรายจ่ายอื่นที่มีอยู่แล้ว — **ไม่ใส่ unique constraint บน `doc_no`**
เพราะกลุ่ม 11 ชุดที่ยืนยันแล้วว่าถูกต้อง (เอกสารรวมหลายรายการ เช่น ค่าจ้างเหมายานพาหนะ +
ค่าส่งไปรษณีย์ ในเอกสารเดียวกัน) ต้องยังบันทึกเป็นคนละแถวได้ — เตือนอย่างเดียวให้เจ้าหน้าที่ตัดสินใจเอง

## เหตุผลที่รวมเฉพาะ 3-key match (ไม่รวมกว้างกว่านี้)

- รวมกว้างกว่านี้ (เช่นแค่ `doc_no` ตรงกัน) จะไปรวมกลุ่ม 11 ชุดที่ถูกต้องอยู่แล้วเข้าด้วยกันผิดๆ
- รวมแคบกว่านี้ (ต้องมนุษย์ยืนยันทุกครั้ง) ทำให้ import ไฟล์ใหญ่ไม่ scale — ใช้แรงงานคนเท่าเดิม
- 3 คีย์ต้องตรงกันทุกตัวอักษรถือว่าเข้มงวดพอที่จะไม่รวมผิดโดยบังเอิญ (description เป็น free text
  ที่เจ้าหน้าที่พิมพ์เอง ไม่ใช่ enum ที่จะบังเอิญตรงกันได้ง่าย)

## ผลที่ตามมา

- Import ไฟล์รายจ่ายซ้ำหลังแก้โค้ดนี้ (ทั้งที่ DB มีแถวเก่าแบบแตกอยู่แล้ว) จะชน trigger
  `allocation_not_exceed` เพราะ dedup query เดิมจะไม่ match (ยอดต่างกัน) — ล้มเหลวอย่างปลอดภัย
  ด้วยข้อความ P0001 ภาษาไทยที่อ่านรู้เรื่อง ไม่ใช่ import ซ้ำเงียบๆ
- 14 ชุด "บจ." ที่ค้างจาก session ก่อนยังไม่ได้แตะ — คนละประเด็นกับ ADR นี้ (เป็นข้อมูลเก่าที่มีอยู่แล้ว
  ไม่ใช่ pipeline ที่จะสร้างซ้ำ) รอผู้ใช้ตัดสินใจแยกต่างหาก
