# ADR-008: แก้บั๊กค้นหา — ประกอบ `.or()` ด้วยการ quote ค่า ไม่ใช่ตัดอักขระทิ้ง

สถานะ: ยอมรับแล้ว (18 ก.ค. 2569)

## บริบท

ผู้ใช้แจ้ง production bug 3 อาการที่ดูไม่เกี่ยวกัน: (1) ค้นหาเลขที่ใบเสร็จที่มีจุด (`บส.16`) ไม่เจอ
ข้อมูล (2) หน้ารายจ่ายไม่ยอมเลือกใบเสร็จที่เพิ่งเพิ่ม (3) มีใบเสร็จ `บส.31/2200` จริง แต่ตัดเงินจาก
ใบนี้ไม่ได้ ("ไม่พบใบเสร็จที่มียอดคงเหลือตรงกับ...") — ทั้งสามอาการมาจากจุดเดียวกัน

## ต้นเหตุ

`sanitizeSearchTerm()` (เพิ่มใน ADR-005/security รอบ 1, commit `afdc8c1`) ตัดอักขระ `, ( ) . *`
ทิ้งจากคำค้นหาทั้งหมดก่อนต่อเป็น string ของ PostgREST `.or()`:

```
"บส.16"  --sanitize-->  "บส16"  --ilike %บส16%-->  ไม่ match "บส.16/4433"
```

เลขที่ใบเสร็จ/เลขที่ส่งออกจริงของระบบนี้มีจุดเสมอ (`บส.31/2200`, `ท.13/69`, `บจ.10/57`) การตัดจุด
ทิ้งจึงทำให้ค้นด้วยเลขที่เต็มไม่เจอเลย **ADR-007 ข้อ "non-issue ที่ตรวจแล้วปลอดภัย" เคยยืนยันผิดว่า
`sanitizeSearchTerm` ของรอบ 1 ยังใช้งานได้ถูกต้อง — บรรทัดนี้ถือเป็นโมฆะ แทนที่ด้วย ADR นี้**

## เหตุผลเดิมผิดตรงไหน — ยืนยันด้วยการยิง PostgREST จริง (read-only)

คอมเมนต์เดิมอ้างว่า `.` เป็น separator ของ `.or()` (`column.operator.value`) จึงต้องตัด
**ซึ่งไม่จริง** — ยิง query จริงไปที่ Supabase production (อ่านอย่างเดียว เช็คแค่ syntax ผ่าน/ไม่ผ่าน):

| กรณี | HTTP | สรุป |
|---|---|---|
| ไม่ quote + มีจุด/ทับ `*บส.31/2200*` | 200 | `.` `/` ไม่เคยทำ syntax พัง |
| quote + มีจุด/ทับ `"%บส.31/2200%"` | 200 | quote ใช้ได้ |
| quote + มี comma/วงเล็บ `"%a, b (x)%"` | 200 | quote กันอักขระอันตรายได้จริง |
| **ไม่ quote + มี comma** `*a,b*` | **400** (PGRST100) | `,` อันตรายจริง — คั่นเงื่อนไข |

สรุป: อักขระที่อันตรายจริงคือ `,` (คั่นเงื่อนไข) และวงเล็บ (จัดกลุ่ม) **ไม่ใช่ `.`** วิธีที่ถูกต้อง
คือครอบค่าด้วย `"` แล้ว escape `\"` `\\` ตามที่ PostgREST รองรับ ไม่ใช่ตัดอักขระทิ้ง — ได้ผลข้างเคียง
ดีขึ้นด้วย: ชื่อบริษัทที่มีวงเล็บ (`... จำกัด (มหาชน)`) ค้นเจอแล้ว ซึ่งของเดิมค้นไม่เจอเช่นกัน

## การแก้ไข

1. **`src/lib/search.ts`**: แทน `sanitizeSearchTerm()` ด้วย `orIlikeFilter(columns, raw)` — ครอบค่า
   ด้วย `"` และ escape `\`/`"` แล้วประกอบเป็น `.or()` filter string ให้ครบทุกคอลัมน์ในที่เดียว
   (ลดโอกาสพลาดซ้ำที่แต่ละหน้าต้องต่อ string เอง)
2. อัปเดต 3 จุดที่เรียกใช้: `donations/page.tsx`, `expenses/page.tsx`, `expense-form.tsx`
   (ค้นหาใบเสร็จเพื่อตัดเงิน — คือจุดที่ทำให้อาการ 2/3 เกิด)
3. **`src/lib/demo/engine.ts`**: ตัว parser ของ `or` เดิมทำ `raw.split(",")` ตรงๆ ซึ่งพัง 2 ทาง
   เมื่อค่าถูก quote: (ก) `,` ในค่าที่ quote แล้วถูกตัดกลางคำ (ข) ค่าที่ได้ติด `"` ค้างมาด้วย
   แยก parser ออกเป็นโมดูลใหม่ `src/lib/demo/or-filter.ts` (`splitOrTopLevel` + `unquoteValue`)
   เพื่อให้ทดสอบตรงได้โดยไม่ต้อง stub `"server-only"` — เป็นบั๊กคลาสเดียวกับที่ ADR-007 เตือนไว้
   ท้ายไฟล์ (demo engine drift) ถ้าไม่แก้คู่กัน demo mode จะให้ผลต่างจากของจริงอีกรอบ

## การตรวจสอบ

- Unit test ใหม่ 16 เคส (`search.test.ts`, `or-filter.test.ts`) ครอบคลุมเลขที่มีจุดจริง
  (`บส.31/2200`, `ท.13/69`), escape `"`/`\`, และ round-trip ระหว่าง `orIlikeFilter` กับ parser
  ของ demo engine — ของเดิมเขียนเทสต์ยืนยันสมมติฐานที่ผิดไว้เอง (`x.y` ต้องกลายเป็น `xy`) และใช้เคส
  `ท13/69` ที่ไม่มีจุด จึงไม่มีเทสต์ไหนจับบั๊กนี้ได้เลย
- ยิง filter string ที่โค้ดจริงสร้าง (ทั้ง 3 เคส: จุดในใบเสร็จ, จุดในเลขที่ส่งออก, comma+วงเล็บ)
  เข้า PostgREST production จริงผ่าน `.or()` แบบเดียวกับที่ supabase-js ส่ง (ครอบ `(...)` รอบนอก)
  — ได้ HTTP 200 ครบ (อ่านอย่างเดียว ไม่มีการเขียนข้อมูล)
- Browser demo mode: ค้นหา `ท.5/68` ที่หน้ารายจ่าย → เจอ 1 รายการตรงเป๊ะ (เดิมค้นแบบมีจุดไม่เจอเลย);
  เปิดฟอร์มเพิ่มรายจ่ายค้นหา `RC1030003` → เจอและแสดงยอดคงเหลือถูกต้อง (path เดียวกับอาการ 2/3)
- `npm run test` (71 ผ่าน) + `npm run build` ผ่าน

## Non-issue ที่ตรวจเพิ่มระหว่างทาง

`vercel.json` เหลือแค่ `regions` (ไม่มี security headers) ไม่ใช่ regression — ย้ายไป
`next.config.ts` `headers()` แล้วตาม ADR-007 ข้อ 5 ตรวจยืนยันแล้วว่า CSP/HSTS ยังอยู่ครบ
